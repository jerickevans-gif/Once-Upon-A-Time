01

Methodology

How this audit was conducted — the tools, the AI agents, the decisions, and the loops. A case study other designers can replicate.

Operator: Jerick Evans, UX Designer (Tech Fleet, Phase 4 design process for Once Upon A Time, a Queens-based nonprofit arts-education organization). My role was to audit the production site against the original PDF design set, surface every gap, decide which gaps mattered, and ship the fixes — before the team migrates the codebase into a Shopify theme.

Surface area: 51 HTML pages, ~7,000 lines of CSS / JS, four heavy script systems (mock-system, auth-modal, site, service-worker), 35 reference PDFs in the design source folder. Manual click-through across all of that would have taken a week. With AI agents conducting the survey work in parallel, the same audit fits in an afternoon.

01 · 08

Tools & Environment

  • Claude Code (Opus 4.7, 1M context) as the conductor — running locally, with file-edit, shell, and image-read tools.
  • Headless Chrome for live-rendered screenshots of every page state, so visual verification is replicable instead of "looks fine on my machine".
  • Python + sed for site-wide find/replace passes that touch dozens of files at once with diff-able output.
  • Git + GitHub Pages as the deploy substrate — every audit finding became a commit; every commit produced a re-deployable URL.
  • ffmpeg / Pillow for image compression and screenshot cropping.
02 · 08

The Agent Pattern

Claude Code lets you spin up sub-agents for independent research tasks. I used three distinct agent types deliberately:

  • Explore (read-only). Crawls the codebase to answer scoped questions. Used for the four parallel audits — link integrity, header/footer consistency, form/handler wiring, accessibility.
  • general-purpose. Can write code. Used when an audit produced an actionable fix that needed bulk edits.
  • Plan. Architectural design. Reserved for the bigger structural decisions (footer canonicalization, sync tool design).
03 · 08

Parallel-Audit Workflow

Each pass dispatched multiple Explore agents at once with carefully scoped prompts. The audit that produced this inventory was four parallel agents firing in a single round:

  1. Link-integrity audit (every href, img src, url())
  2. Cross-page consistency audit (header markup drift across 51 files)
  3. Form / button / data-mock wiring audit
  4. Accessibility audit (alt, labels, contrast, heading hierarchy, focus, dialogs)

Each agent returned a 400-word findings report. Total wall-clock: ~3 minutes. A traditional QA round on a 51-page site is ~2 days.

04 · 08

Prompt Scoping

  • Briefed like a colleague who just walked in. Each agent gets the project context, the priors I already ruled out, and what specifically to look for — never a one-line "audit this".
  • Capped output length. Every prompt ends with "cap report at 500 words" so context costs stay bounded.
  • Asked for specifics, not summaries. "Per finding: file:line + one-sentence explanation" beats "discuss accessibility issues".
  • Excluded healthy categories. "If a class is fine, say so in one line — don't enumerate every link." Signal stays high.
05 · 08

The Triage Loop

Every audit finding ran through the same three-question filter:

  1. Is it broken in production? If yes, fix it now.
  2. Is it consistent across the site? If no, normalize it.
  3. Did the design PDFs cover it? If no, log it in this inventory with a design-note for the team.

The first two categories produce shippable commits. The third produces this deck.

06 · 08

Verification Protocol

  • Headless Chrome screenshots at multiple viewports (1280, 1440, 375 mobile) before and after every change.
  • cURL the served HTML to confirm GitHub Pages is shipping what's in the local repo.
  • Hard-refresh in a fresh user-data-dir to bypass the service-worker cache when visual state didn't match source.
  • Re-ran the same audit prompts after fixes landed — "0 findings" was the success signal.

This last step caught the highest-impact bug: a JS-injected duplicate search bar that the static-HTML cleanup didn't address. The screenshot showed two search inputs, the served HTML had one, the diff was a runtime injection in mock-system.js.

07 · 08

"Off-Spec" Criteria

An item entered this inventory if it satisfied both:

  1. It exists and ships in production on the live URL.
  2. No PDF in PDF Finals/ shows it directly, OR the PDF shows it ambiguously enough that a designer would draft a fresh component before re-implementation.

Importance ranking (1–5) is judged against blast radius — does removing this item break a journey, or just polish?

08 · 08

Why This Matters

Traditional UX audits are bounded by how much a designer can click through manually. Coverage gaps hide in long-tail pages (404, offline, password reset, transactional confirmations) and in cross-cutting patterns (toasts, modals, cookie banners) that only manifest under specific conditions.

With AI agents handling the survey, the designer's role shifts upstream: writing better prompts, ranking findings, making the calls about what to spec, what to keep, what to drop. The agent does the click-through; the designer does the design.

The end-to-end loop, one example

  1. Inspect. Spin up Explore agent → "Audit programs.html for any feature not in the PDF design".
  2. Triage. Agent reports the chip-filter row, the in-page search, the bookmark/share icons. None are in the PDF.
  3. Decide. All three are net-positive UX. Keep — but flag for design spec.
  4. Implement. Either keep as-is, or refactor to match a new spec when the design team produces one.
  5. Verify. Headless screenshots at 1280 and 375. Diff against PDF reference image. Push to live.
  6. Document. Log to audit/inventory.json with importance and design notes.

Total elapsed time per item: 3–8 minutes. A 30-item inventory in a single afternoon, with every item screenshot-verified, ranked, and ready to hand to design.

02

UX framework that guided the audit

The audit ran on heuristics, not vibes. The framework comes from a custom Claude skill the operator built and brought into this engagement.

Source: ux-ui-audit — a Claude Code skill Jerick wrote and distilled from a real audit of EigenAid, a voice-first crisis-knowledge PWA. The skill encodes the principle: "can I find and act on this in 3 seconds, in stress, with one hand?" Every audit prompt in this engagement was scoped through that lens, then expanded against the OAUT-specific surface area. The skill is the reason the agent hunted for emergency-style usability gaps instead of just running generic Lighthouse checks.

Emergency usability

Find & act in 3 seconds. Single-tap primary action. No deep menu trees.

Audit applied to OUAT Header CTA → enrollment in <2 clicks; cart visible at every viewport; sign-in modal opens without page navigation.

WCAG 2.1 AA

Skip links, focus rings, alt text, contrast ≥ 4.5:1 normal text, heading hierarchy, dialog roles.

Audit applied to OUAT Caught garden chip at 3.68:1 (failed). Caught h2→h4 skip in 38 page footers. 0 a11y findings remaining.

Responsive breakpoints

Verify the experience at 375 (mobile), 768 (tablet), 1280 (desktop), 1440 (wide). Don't trust source — render and look.

Audit applied to OUAT Headless Chrome at all four widths. Caught header search icon hidden on mobile — flagged for hamburger-menu fix.

Search & discovery

Every page has a path to global search. In-page filters complement, never replace, the global entry point.

Audit applied to OUAT Built header search icon-link on 47 pages. Built programs filter that doesn't shadow the global SERP.

Error & offline

Reliable fallback for every failure mode — 404, 500, maintenance, offline. Branded, recoverable, never a dead end.

Audit applied to OUAT Service worker + offline.html / 500.html / maintenance.html, all with shared header / footer for recovery routes.

Emotional & trust design

Tone, micro-copy, voice. Toasts that sound human. Confirmations that don't shame. Empty states that invite, not nag.

Audit applied to OUAT 25 data-mock toasts written in plain language; cookie banner uses "small number of essential cookies" not legalese; "Keep / Spec / Replace" verdict pills (this deck) instead of pass/fail.

Sample prompts the operator wrote

Real audit prompts from this engagement. Every one ends with a length cap and a structured-output requirement — that's how high-signal reports come back from a sub-agent.

Prompt 01 · Parallel link-integrity audit
# Explore agent (read-only)
Audit /Users/.../Once-Upon-A-Time/ for broken internal links. Static HTML/CSS/JS site (51 *.html files), GitHub Pages, Shopify-bound.

For each broken reference report a single line: <source-file>:<line> → <missing-target>.

Also flag two specific cases I want to know about: any remaining href="index.html#programs" (was renamed — should be 0); any data-formspree-id="REPLACE_ME" placeholders.

# Cap report at 350 words. If something is OK across the board, say so in one line — don't list every healthy link.
Prompt 02 · Off-spec inventory
# Explore agent — produces this deck's data
Produce a categorized inventory of "things that exist in the repo but have no obvious PDF source" so I can build a branded design-handoff document from your findings.

For each item: file path(s), 1-line description, importance ranking 1–5 where 5 = core to the site functioning at all, 1 = minor copy.

Output as JSON-style structured data. Aim for 25–40 items total. Don't enumerate every utility; combine similar items.

# Cap at 1500 words. After the JSON, add a 3–5 paragraph narrative summary I can drop into the intro of my design-team presentation.
Prompt 03 · Cross-page consistency drift
# Explore agent (read-only)
Audit for cross-page drift in the shared header/nav and footer markup. After mass sed-based edits I want to verify nothing got missed.

What "consistent" should look like: <a href="programs.html">Programs</a> in primary-nav; search icon between Contact and Cart in header-actions.

For each page, flag deviations — missing search icon, missing/mis-ordered nav links, different aria-labels, different cart badge counts, footer column inconsistency.

# Output: per-issue line "<file>: <one-sentence>". If a class of files is fine, say "Nav: 47/51 match canonical, 4 deviations below".
03

Human ↔ AI loop

The deck would be dishonest without showing what the operator caught and corrected. Four moments where the human-in-the-loop materially improved the output.

Moment 01

"I am still seeing 2 search bars."

Operator: Pushed back when the static-HTML cleanup didn't fix what the screenshot still showed.

Agent: Initially insisted the served HTML was clean (because it was). On re-prompt, ran a JS-injection grep — found mock-system.js auto-prepending a search input to every .header-actions.

Outcome: Removed the JS injection, restored the static icon-link, kept the in-page filter. Single source of truth for the search affordance on every page.

Moment 02

"The bottom nav looks off. Consult the PDFs."

Operator: Caught that "About / Programs / Instructors" + "Events / Support Us / Volunteer" didn't match the design intent (PDF showed a leaner two-link-per-column footer).

Agent: Counted 8 distinct footer variants across 47 pages. Built one canonical block matching the PDF, plus tools/sync-footer.py to keep all 47 in lock-step.

Outcome: Idempotent footer. "0 file(s) would change" on every dry-run. Migration to Shopify sections/footer.liquid becomes a 5-minute port.

Moment 03

"Look at the about page — these screenshots are what these sections should look like."

Operator: Provided two PDF reference screenshots and pointed at specific sections, not vague feedback. Gave me what to compare to.

Agent: Read the existing markup, diff'd against the screenshots — Leadership needed left-aligned hero typography + "[Add Title here]" placeholder; Success Stories needed a horizontal carousel with 160px circular portraits, not a 3-column card grid.

Outcome: Both sections rebuilt to match the PDF in one pass. The reference image is the assertion test.

Moment 04

"Ideally there is a global bottom nav."

Operator: Named the underlying architectural ask, not just the surface symptom.

Agent: Static GitHub Pages can't truly include a partial. Offered three options (HTML find/replace today, JS injection, Liquid section post-Shopify) with trade-offs around SEO, build complexity, and maintainability.

Outcome: Operator chose option 1 + commit the sync script as a stopgap. Every option had a clear owner and a defensible reason. The agent didn't make the call alone.

The screenshot is the assertion test. The PDF is the source of truth. The agent does the diff. — Working pattern, this engagement

08

Out of scope — owned by Shopify

Things the design team does not need to spec because Shopify (or its standard apps) handles them natively. This list exists so designer time goes to the surfaces that actually need design work — not patterns the platform owns.

Why this list matters

If we design something Shopify already handles, two things happen: (1) the design gets discarded during dev because it conflicts with the native pattern, or (2) the design gets implemented and breaks when Shopify rolls a platform update. The boundaries below are firm.

For each surface, we note what Shopify owns and where you can still influence the look. Most categories allow some customisation via Liquid sections, app blocks, or Checkout extensibility — but the core flow is platform.

Cart drawer

Owned: Shopify's native cart drawer (Dawn theme + section block). Item rows, qty steppers, subtotal, "go to checkout" CTA.

You can still spec: the cart drawer's empty state, free-shipping progress bar, gift-message UI, suggested-products module.

Checkout

Owned: Shopify Checkout (Plus required for full extensibility). Address forms, payment fields, tax/shipping calculation, order confirmation email.

You can still spec: checkout header logo, banner upsells, custom fields (allergy info), thank-you page extensions, branded post-purchase pages.

Customer accounts

Owned: Shopify Customer Account (the new account UI). Order history, addresses, password, 2FA, magic-link sign-in.

You can still spec: the account-area homepage layout, custom widgets (next-class card, child profiles), branded sidebar, Class History as an extension.

Search & discovery

Owned: Shopify Search & Discovery app. Indexing, ranking, filters, synonyms, search analytics. Replaces the in-house search.html entirely.

You can still spec: SERP layout, suggested-query chips, recent-search history, branded empty state. The data flow is platform; the visual is yours.

Email

Owned: Shopify Email or Klaviyo for transactional + marketing. Order confirmation, receipt, password reset, abandoned cart, weekly digest. All templated by the email platform.

You can still spec: email header / footer brand bar, content modules (Featured Class, Donor Spotlight), tone of voice for transactional copy.

Inventory states

Owned: Shopify product inventory (in-stock / sold-out / waitlist). Storefront automatically reflects whatever inventory states the merchant configures.

You can still spec: the visual treatment of each state (chip colour, copy: "Filling fast", "Waitlist open", "Closed"), back-in-stock email signup form.

Tax / payment / compliance

Owned: Tax calculation (Shopify Tax), payment processing (Shopify Payments + 3rd-party gateways), PCI compliance, fraud analysis, refund flows.

You can still spec: nothing direct here. This is platform infrastructure. Don't waste design time on it.

Cookie consent / GDPR

Owned: Shopify's Cookie Banner (built into Online Store 2.0) or 3rd-party app like Pandectes. Categorised consent, preference centre, audit log.

You can still spec: the banner copy / tone, preference-centre layout, the consent-acknowledgement micro-toast.

Sitemap / robots / SEO infra

Owned: Shopify auto-generates sitemap.xml, robots.txt, canonical tags, structured data on product / collection pages, hreflang for multi-language stores.

You can still spec: page-level meta titles / descriptions (CMS), product schema fields (course type, age range), Open Graph imagery for shareable links.

Translations / locales

Owned: Shopify Markets / Translate & Adapt. Multi-language storefront, currency switching, regional pricing.

You can still spec: the language picker UI, locale-specific imagery, US-vs-international copy variations (likely n/a for OUAT but worth flagging).

App marketplace components

Owned: Reviews (Judge.me / Yotpo), wishlist (Smart Wishlist), gift cards, subscriptions (Recharge), live chat (Tidio / Gorgias) all ship with their own UI.

You can still spec: brand styling for each app's customisable surface — most reputable apps expose colour, fonts, copy. Don't redesign their forms; restyle their existing components.

Form validation / error handling

Owned: Shopify Forms (built-in checkout / contact form validation). Field-level errors, required-field handling, server-side validation, native browser hints.

You can still spec: the visual treatment of inline errors (colour, icon, copy tone), validation timing on custom Liquid forms, the global empty-state pattern.

Loading / skeleton states

Owned: Shopify Hydrogen storefronts use built-in Suspense / streaming. Theme storefronts use server-rendered HTML so skeletons aren't load-blocking the same way.

You can still spec: skeleton tokens (background, highlight) for places designers explicitly choose to use them — search results, lazy-loaded sections.

Three decisions per item

Keep. Liquid port the existing implementation as-is. Best for unobjectionable cross-site infra (skip link, footer canonical, JSON-LD).

Spec. Design team writes a Figma component, then engineering rebuilds in Shopify. Best for items at importance 4–5 that ship daily without formal design (auth modal, search, hero carousel, calendar).

Replace. Drop the hand-rolled version in favour of a Shopify-native equivalent. Best for the in-house search system (→ Shopify Search & Discovery), the data-mock interaction layer (→ real backends), parts of form-validation.

Once each of the 30 items has an owner and a verdict, this inventory becomes the migration's checklist.
