Auditing the gap between design and production

The original PDF set delivered roughly 35 page mocks for the Once Upon A Time arts-education site. Production ships 51 pages and a deep catalogue of cross-cutting interactions. This deck is a UX audit of everything that was filled in beyond the original spec — and a case study in how AI coding agents accelerate the audit work that used to take a designer days of click-through.

Methodology

How this audit was conducted — the tools, the AI agents, the decisions, and the loops. A case study other designers can replicate on their own projects.

Operator: Jerick Evans, UX Designer (Tech Fleet, Phase 4 design process for Once Upon A Time, a Queens-based nonprofit arts-education organization). My role was to audit the production site against the original PDF design set, surface every gap, decide which gaps mattered, and ship the fixes — before the team migrates the codebase into a Shopify theme.

Surface area: 51 HTML pages, ~7000 lines of CSS / JS, four heavy script systems (mock-system, auth-modal, site, service-worker), 35 reference PDFs in the design source folder. Manual click-through across all of that would have taken a week. With AI agents conducting the survey work in parallel, the same audit fits in an afternoon.

01

Tools & environment

  • Claude Code (Opus 4.7, 1M context) as the conductor — running locally, with file-edit, shell, and image-read tools.
  • Headless Chrome for live-rendered screenshots of every page state, so visual verification is replicable instead of "looks fine on my machine".
  • Python + sed for site-wide find/replace passes that touch dozens of files at once with diff-able output.
  • Git + GitHub Pages as the deploy substrate — every audit finding became a commit; every commit produced a re-deployable URL the design team can review.
  • ffmpeg for image compression passes (PNG → JPEG q4) when the audit surfaced perf wins.
02

The agent pattern

Claude Code lets you spin up sub-agents for independent research tasks. I used three distinct agent types deliberately:

  • Explore (read-only). Crawls the codebase to answer scoped questions. Used for the four parallel audits — link integrity, header/footer consistency, form/handler wiring, accessibility.
  • general-purpose. Can write code. Used when an audit produced an actionable fix that needed bulk edits.
  • Plan. Architectural design. Reserved for the bigger structural decisions (e.g. consolidating eight footer variants into one canonical block, the canonical-footer sync tool).
03

Parallel-audit workflow

For each pass I dispatched multiple Explore agents at once with carefully scoped prompts. The audit that produced this inventory was four parallel agents firing in a single round:

  1. Link-integrity audit (every href, img src, url())
  2. Cross-page consistency audit (header markup drift across 51 files)
  3. Form / button / data-mock wiring audit
  4. Accessibility audit (alt, labels, contrast, heading hierarchy, focus, dialogs)

Each agent returned a 400-word findings report. Total wall-clock: about 3 minutes. A traditional QA round on a 51-page site is ~2 days.

04

How I scoped each prompt

  • Briefed like a colleague who just walked into the room. Each agent gets the project context, the priors I already ruled out, and what specifically to look for — never a one-line "audit this".
  • Capped output length. Every prompt ends with "cap report at 500 words" so context costs stay bounded.
  • Asked for specifics, not summaries. "Per finding: file:line + one-sentence explanation" beats "discuss accessibility issues".
  • Excluded healthy categories. "If a class is fine, say so in one line — don't enumerate every link" keeps signal high.
05

The triage loop

Every audit finding ran through the same three-question filter:

  1. Is it broken in production? If yes, fix it now (newsletter nav link → wrong URL, garden chip → 3.68:1 contrast).
  2. Is it consistent across the site? If no, normalize it (eight footer variants → one; programs nav drift → parity restored).
  3. Did the design PDFs cover it? If no, log it in this inventory with a design-note for the team.

The first two categories produce shippable commits. The third produces this deck.

06

How I verified the fixes

  • Headless Chrome screenshots at multiple viewports (1280, 1440, 375 mobile) before and after every change.
  • cURL the served HTML to confirm what GitHub Pages is actually shipping vs. what's in the local repo.
  • Hard-refreshed in a fresh user-data-dir to bypass the service-worker cache when the visual state didn't match the source.
  • Re-ran the same audit prompts after fixes landed, looking for "0 findings" as the success signal.

This last step caught the JS-injected duplicate search bar that the static-HTML cleanup didn't address — the screenshot showed two search inputs, the HTML showed one, the diff was a runtime injection in mock-system.js.

07

Decision criteria for "off-spec"

An item entered this inventory if it satisfied both:

  1. It exists and ships in production on the live URL.
  2. No PDF in PDF Finals/ shows it directly, OR the PDF shows it ambiguously enough that a designer would draft a fresh component before re-implementation.

Importance ranking (1–5) is judged against blast radius — does removing this item break a journey, or just polish? Fives are critical infrastructure (auth modal, search). Twos are tooling (sitemap, JSON-LD).

08

Why this matters for design teams

The traditional UX audit is bounded by how much a designer can click through manually. Coverage gaps hide in long-tail pages (404, offline, password reset, transactional confirmations) and in cross-cutting patterns (toasts, modals, cookie banners) that only manifest under specific conditions.

With AI agents handling the survey, the designer's role shifts upstream: writing better prompts, ranking findings, and making the calls about what to spec, what to keep, what to drop. The agent does the click-through; the designer does the design.

The end-to-end loop, one example

  1. Inspect. Spin up Explore agent → "Audit programs.html for any feature not in the PDF design".
  2. Triage. Agent reports the chip-filter row, the in-page search, the bookmark / share icons. None of these are in the PDF.
  3. Decide. All three are net-positive UX. Keep — but flag for design spec.
  4. Implement. Either keep as-is, or refactor to match a new spec when the design team produces one.
  5. Verify. Headless screenshots at 1280 and 375. Diff against PDF reference image. Push to live.
  6. Document. Log to audit/inventory.json with importance and design notes.

Total elapsed time per item: 3–8 minutes. A 30-item inventory in a single afternoon, with every item screenshot-verified, ranked, and ready to hand to design.

Three decisions per item

Keep. Liquid port the existing implementation as-is. Best for unobjectionable cross-site infra (skip link, footer canonical, JSON-LD).

Spec. Design team writes a Figma component, then engineering rebuilds in Shopify. Best for items at importance 4–5 that ship daily without formal design (auth modal, search, hero carousel, calendar).

Replace. Drop our hand-rolled version in favour of a Shopify-native equivalent. Best for the in-house search system (→ Shopify Search & Discovery), the data-mock interaction layer (→ real backends), and parts of the form-validation pattern.

Once each of the 30 items has an owner and a verdict, this inventory becomes the migration's checklist.
